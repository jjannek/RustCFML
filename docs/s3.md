# S3 (and S3-Compatible) Object Storage

RustCFML ships with first-class support for AWS S3 and S3-compatible providers (Cloudflare R2, MinIO, DigitalOcean Spaces, Backblaze B2, Wasabi, etc.). Two APIs are available, both backed by the same client cache and credential chain:

1. **Explicit `S3*` builtins** (`S3Read`, `S3Write`, `S3ListBucket`, ŌĆ”) ŌĆö Lucee-compatible function signatures.
2. **Transparent `s3://` VFS** ŌĆö pass an `s3://bucket/key` URL to any normal CFML file/directory function (`fileRead`, `fileWrite`, `directoryList`, etc.) and it routes through S3 automatically.

S3 support is part of the standard `rustcfml` binary ŌĆö no special build flags. (The dependency adds the AWS SDK; the WASM target deliberately excludes it ŌĆö see [WASM / Cloudflare Workers](#wasm--cloudflare-workers) below.)

---

## Quick Start

```cfml
<cfscript>
// Explicit form
s3Write("my-bucket", "logs/today.txt", "hello world");
content = s3Read("my-bucket", "logs/today.txt");
listing = s3ListBucket("my-bucket", "logs/");

// Transparent VFS ŌĆö identical effect:
fileWrite("s3://my-bucket/logs/today.txt", "hello world");
content = fileRead("s3://my-bucket/logs/today.txt");
listing = directoryList("s3://my-bucket/logs/");
</cfscript>
```

Credentials come from one of (in priority order): inline function args, the URL itself, `Application.cfc` `this.s3 = {...}`, or environment variables (see [Credentials](#credentials)).

## Available Functions

### Explicit S3 builtins

| Function | Signature | Notes |
|---|---|---|
| `S3Read` | `(bucket, key [, charset, accessKeyId, secretKey, host])` | Returns the body as a UTF-8 string |
| `S3ReadBinary` | `(bucket, key [, accessKeyId, secretKey, host])` | Returns `Binary` |
| `S3Write` | `(bucket, key, value [, charset, mimeType, acl, location, accessKeyId, secretKey, host])` | Auto-detects MIME from key extension |
| `S3Upload` | `(bucket, key, sourcePath [, acl, location, accessKeyId, secretKey, host])` | Reads from local file |
| `S3Download` | `(bucket, key [, targetPath, charset, accessKeyId, secretKey, host])` | No `targetPath` ŌåÆ returns String |
| `S3Exists` | `(bucket [, key, accessKeyId, secretKey, host])` | No `key` ŌåÆ tests bucket |
| `S3Delete` | `(bucket [, key, force, accessKeyId, secretKey, host])` | No `key` ŌåÆ deletes bucket (`force=true` empties first) |
| `S3Copy` | `(srcBucket, srcKey, dstBucket [, dstKey, acl, accessKeyId, secretKey, host])` | |
| `S3Move` | `(srcBucket, srcKey, dstBucket [, dstKey, acl, accessKeyId, secretKey, host])` | Copy + delete source |
| `S3ListBucket` | `(bucket [, prefix, maxKeys, accessKeyId, secretKey, host])` | Returns array of structs `{key, size, lastModified, eTag, storageClass, isDirectory}` |
| `S3ListBuckets` | `([accessKeyId, secretKey, host])` | Array of `{bucketName, creationDate}` |
| `S3CreateBucket` | `(bucket [, acl, location, accessKeyId, secretKey, host])` | |
| `S3ClearBucket` | `(bucket [, accessKeyId, secretKey, host])` | Deletes every object |
| `S3GetMetaData` | `(bucket, key [, ...])` or `(s3:// URL [, ...])` | Returns struct with `content_type`, `content_length`, `etag`, `last_modified`, `storage_class`, `x-amz-meta-*` |
| `S3GeneratePresignedURL` | `(bucket, key [, expiresMinutes=60, method="GET", accessKeyId, secretKey, host])` | |
| `S3GenerateURI` | `(bucket [, key, style="virtualhost", secure=true, host])` | Pure ŌĆö no network call |
| `StoreGetMetadata` | `(s3:// URL)` | Convenience alias |

### Transparent `s3://` VFS

These CFML built-ins automatically route to S3 when the path argument is an `s3://...` URL:

```
fileRead          fileReadBinary   fileWrite        fileAppend
fileExists        fileDelete       fileCopy         fileMove
directoryList     directoryExists  directoryCreate  directoryDelete
directoryRename   directoryCopy
```

`directoryList` with `recurse=false` (default) returns objects under the prefix bounded by `/` delimiter (S3 "common prefixes" become `isDirectory=true` entries). With `recurse=true` it returns every key under the prefix.

The optional third argument controls the return shape:
- `path` (default) ŌåÆ array of `s3://...` URI strings
- `query` ŌåÆ array of structs with full metadata

## URL Format

```
s3://[accessKey:secretKey@[host/]]bucket[/key]
```

| Example | Effect |
|---|---|
| `s3://my-bucket/photo.jpg` | Uses creds from `Application.cfc`/env |
| `s3://KEY:SECRET@my-bucket/photo.jpg` | Inline AWS creds, default endpoint |
| `s3://KEY:SECRET@account.r2.cloudflarestorage.com/my-bucket/photo.jpg` | R2 endpoint |
| `s3://KEY:SECRET@minio.example.com/my-bucket/photo.jpg` | MinIO |

The runtime treats the segment after `@` as a host **iff** it contains a `.` *and* inline creds are present; otherwise it's the bucket name.

> **Inline-credential URLs and function args bypass the configured key prefix** (see below) ŌĆö they're treated as explicit addressing.

## Credentials

Resolution order (first non-empty wins):

1. **Inline function arguments** ŌĆö `accessKeyId`, `secretKey`, `host` positional args.
2. **Inline URL credentials** ŌĆö `s3://key:sec@host/bucket/key`.
3. **`Application.cfc this.s3` struct.**
4. **Environment variables.**

### `Application.cfc` form

```cfml
component {
    this.name = "myapp";
    this.s3 = {
        accessKeyId:     "AKIAŌĆ”",
        awsSecretKey:    "abcdŌĆ”",     // or secretKey
        defaultLocation: "us-east-1", // bucket region ŌĆö use "auto" for R2
        host:            "https://account.r2.cloudflarestorage.com", // optional
        keyPrefix:       "myapp/"     // optional ŌĆö see Key-Prefix Scoping
    };
    // Optional s3-targeted CFML mappings:
    this.mappings["/logs"] = "s3://KEY:SEC@my-bucket/logs/";
}
```

### Environment variables

The runtime accepts standard AWS names alongside the Lucee-compatible aliases:

| Concept | Variables (checked in order) |
|---|---|
| Access key | `AWS_ACCESS_KEY_ID`, `LUCEE_S3_ACCESSKEYID`, `LUCEE_S3_ACCESSKEY` |
| Secret key | `AWS_SECRET_ACCESS_KEY`, `LUCEE_S3_SECRETACCESSKEY`, `LUCEE_S3_SECRETKEY` |
| Region | `AWS_DEFAULT_REGION`, `AWS_REGION`, `LUCEE_S3_LOCATION`, `LUCEE_S3_REGION` |
| Endpoint URL | `LUCEE_S3_HOST`, `LUCEE_S3_SERVER`, `AWS_ENDPOINT_URL` |
| Key prefix | `RUSTCFML_S3_KEY_PREFIX`, `LUCEE_S3_KEYPREFIX`, `LUCEE_S3_KEY_PREFIX` |

IAM instance profiles and the rest of the standard AWS credential chain are honored automatically when no override is supplied (via `aws-config`).

## Key-Prefix Scoping

Set a key prefix to confine every S3 operation to a subpath of a bucket ŌĆö useful when several apps share a single bucket.

```cfml
// Application.cfc
this.s3 = { accessKeyId: "ŌĆ”", secretKey: "ŌĆ”", keyPrefix: "myapp/" };
```

```cfml
// In your code:
s3Write("shared-bucket", "config/db.json", body);
// Physically writes to:  shared-bucket/myapp/config/db.json

s3Read("shared-bucket", "config/db.json");
// Reads from:            shared-bucket/myapp/config/db.json
//                        ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
//                        but the script never sees the prefix.

fileWrite("s3://shared-bucket/uploads/x.png", body);
// Same ŌĆö VFS form is scoped too.

S3ListBucket("shared-bucket");
// Returns only entries under myapp/, with the prefix stripped from each key.
```

The prefix is **suppressed** when the caller is being explicit:

- Inline credentials in the URL: `s3://key:sec@host/bucket/key`
- Inline credentials as function args: `s3Read(bucket, key, charset, accessKeyId, secretKey)`
- Paths resolved via an `s3://` CFML mapping (the mapping target already encodes the desired prefix)

This makes "escape hatches" possible ŌĆö call the same bucket without prefix scoping when you really need to.

---

## CFML Mappings ŌåÆ `s3://`

`Application.cfc` mappings can target s3:// URLs. The transparent VFS rewrites any path matching the mapping prefix:

```cfml
component {
    this.mappings["/logs"] = "s3://KEY:SECRET@my-bucket/logs/";
}
```

```cfml
fileWrite("/logs/today.txt", "hi");
// Internally rewritten to: s3://KEY:SECRET@my-bucket/logs/today.txt
```

Mapping-rewritten paths bypass the app's `keyPrefix` (the mapping target IS the scoping).

## Local testing with MinIO

The harness at `/tmp/rustcfml-s3-harness/` (not checked into the repo ŌĆö see [README.md](../README.md)) drives a full live test against any S3-compatible endpoint. To spin up a local MinIO and run it:

```bash
docker run -d --name rustcfml-minio \
  -p 9100:9000 -p 9101:9001 \
  -e MINIO_ROOT_USER=rustcfml \
  -e MINIO_ROOT_PASSWORD=rustcfml-secret \
  minio/minio server /data --console-address ":9001"

docker run --rm --network host --entrypoint sh minio/mc -c "
  mc alias set local http://127.0.0.1:9100 rustcfml rustcfml-secret &&
  mc mb -p local/rustcfml-test
"

# .env (in the harness folder) ŌĆö fill in:
#   RUSTCFML_S3_TEST_BUCKET=rustcfml-test
#   AWS_ACCESS_KEY_ID=rustcfml
#   AWS_SECRET_ACCESS_KEY=rustcfml-secret
#   AWS_DEFAULT_REGION=us-east-1
#   AWS_ENDPOINT_URL=http://127.0.0.1:9100
#   RUSTCFML_S3_KEY_PREFIX=harness-app/    # optional

./run_live.sh
```

The harness exercises every explicit function, the transparent VFS, prefix scoping, and mapping shape end-to-end (currently 45 assertions).

## WASM / Cloudflare Workers

**The `s3` cargo feature is _not_ enabled for the WASM target** (`crates/wasm`). The AWS SDK uses tokio + hyper + rustls for transport, none of which compile on `wasm32-unknown-unknown` (the target used by `wasm-pack`, `wasm-bindgen`, and Cloudflare Workers).

Two practical paths if you need S3-style storage from `rustcfml-wasm` / `RustCFMLWorker`:

1. **Use Cloudflare R2 directly via the R2 binding** in your Worker's `wrangler.toml`. This is the native Cloudflare API (no SigV4, no fetch overhead) and is what `RustCFMLWorker` exposes today. The CFML script reads/writes from a host-side binding rather than via the S3 SDK.
2. **Pure-Rust SigV4 + `fetch()`** ŌĆö sign requests with [`aws-sigv4`](https://crates.io/crates/aws-sigv4) (no transport) and send them via the Workers `fetch()` binding. This is a moderate amount of glue code and would need to be plumbed as a separate VM intercept gated on a new `s3-wasm` feature.

The AWS SDK has experimental `wasm32-wasip2` support, but that target isn't available in browsers or Workers, and isn't currently a goal for RustCFML's WASM crate.

If/when path (2) is implemented, the script-facing API (`S3*` functions and `s3://` VFS) will be identical to the native build.

## Architecture notes

- Per-endpoint `aws_sdk_s3::Client` is cached in a process-global `S3Clients` instance (`crates/cfml-stdlib/src/s3.rs`). Each unique `(access_key, region, endpoint, secret_len)` tuple gets one client; client construction is the expensive part (credential chain + HTTPS connector setup).
- All operations are async under the hood. A fallback multi-threaded tokio runtime is created lazily if the caller isn't already inside one (e.g., a CLI script). Inside `--serve` mode / the Workers fetch handler the existing tokio runtime is reused.
- `crates/cfml-vm/src/s3_vfs.rs` hosts the VFS intercept; it runs *before* the sandbox intercept in `call_function`.
- `crates/cfml-stdlib/src/s3_builtins.rs` registers and implements the explicit `S3*` builtins.

See [Compatibility.md](../Compatibility.md) and `tests/s3/test_s3_functions.cfm` for the latest pure (no-network) assertions baked into the binary; live behavior is verified via the external harness.
